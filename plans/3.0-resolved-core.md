# codegraph 3.0 - "Resolved core" (element identity as the substrate)

Supersedes plans/2.1-resolved.md (folds its content in and adds the measured
viability numbers that plan asked for as a Stage 1 gate). Implements pillar P0
of plans/BRD-actuator.md. This is the FIRST step of the actuator redirection:
it changes nothing an agent sees yet, but it is the foundation P1 (conformance)
deepens on and P2 (actuator) cannot exist without.

Mandate: the 2026-07-11 audit's central verdict is that every documented
ceiling (nav resolution stuck at 39/99, wrapper and renamed-parameter ref
reads invisible, callers/callchain name-matched across unrelated same-named
declarations) traces to ONE decision - syntax-only parsing. 3.0 reopens it
deliberately, as an opt-in layer with a zero-setup syntax fallback.

## v3 - no backwards compatibility

This is v3. We do NOT preserve backwards compatibility. The wire format, graph
format version, CLI defaults, and doctrine may all change freely; old
`code_graph.json` files do not need to keep loading (a stale graph rebuilds
itself anyway). This removes the constraints that shaped 2.x (byte-identical
migrations, additive-only stats, old-graph loader warnings) - later stages
change the format outright when it is cleaner to.

## Standing doctrine

ROADMAP standing doctrine applies, with item 1 REPLACED (v3):

OLD: "Syntax-only parsing; no pub get in the host. Ever."
NEW: "Resolved analysis is the DEFAULT. Syntax-only is the automatic zero-setup
fallback (no `.dart_tool/package_config.json`) and the explicit `--syntax`
opt-out, plus the per-file fallback when a single file will not resolve. A
resolved answer must never be less honest than a syntax answer: where
resolution fails for a file, that file falls back to syntax extraction and the
output says so. An explicit `--resolved` with no package_config refuses with a
`pub get` instruction rather than silently degrading."

NEVER-GUESS is unchanged and gets easier: element identity replaces the
name-match allow-lists, so wrong edges from same-named symbols vanish by
construction. Determinism (byte-identical check() output) is unchanged and now
requires sorting element iteration.

## Viability - MEASURED, gate satisfied (2026-07-14)

The 2.1 plan required a cold/warm measurement before Stage 1 could proceed.
Done, via a throwaway spike (AnalysisContextCollection over codegraph's own
26-file pure-Dart package, analyzer 12.1.0, Dart 3.12.1):

- resolved 26/26 files, 0 failures.
- element identity works: 2601/2617 call sites (99%) resolve to a real
  element; cross-file and SDK identity confirmed (`toList` -> `Iterable.toList`
  @ dart:core).
- cost is a one-time warmup plus cheap steady state, NOT linear in the naive
  average: collection setup 175ms; first-file resolve 4083ms (builds the SDK +
  dependency element model once); steady-state per file p50=12ms, p90=63ms.
- corrected extrapolation to a ~1500-file Flutter host: ~4-10s warmup +
  1500 * ~30ms ~= under a minute cold, single-threaded. Well inside the
  "once in a while" build budget.
- memory: ~800MB RSS for 26 files, dominated by the SDK/analyzer element model
  (a near-fixed floor, not per-file growth). This is the daemon's constraint
  (Stage 3), NOT the one-shot build's - the build process exits and reclaims
  it.

Verdict: resolved core is viable for the one-shot build now. Memory is deferred
to the daemon stage. Proceed.

## Wire format (v3 - redesign, not additive)

Per the v3 no-compat rule and plans/knowledge-model.md, the model changes SHAPE,
not just gains keys:
- GraphNode gains `symbolId` - a SCIP-style human-readable resolved symbol
  string (package + path + descriptor, e.g. `fixture/foo/Bar#baz().`), not a
  file:line. Same symbol across packages = same id = cross-package "who uses X"
  for free. `kind` widens beyond file/provider to class/method/field/param.
- GraphEdge gains `confidence`: `resolved` (element identity) | `heuristic`
  (name/token match) | `guessed` (inferred, e.g. nav via a constant). The
  NEVER-GUESS doctrine becomes a queryable column; the actuator refuses to act
  on anything below `resolved` as a WHERE clause, not bespoke code.
- graphFormatVersion bumps freely; pre-v3 graphs do NOT load - the loader
  rejects an old format and triggers a rebuild (a stale graph rebuilds anyway).
  Drop the additive-only / pinned-position contortions from 2.x.
- Determinism KEPT: same source -> same bytes (check() gate, caching). Only
  byte-COMPATIBILITY with the old schema is dropped. Element iteration sorted by
  (symbolId) before emission so resolved extraction stays deterministic.

## Stages (each lands green before the next; ROADMAP item 6)

### Stage 1 - Resolution harness + fallback  [DONE 2026-07-14]
- Resolved is the DEFAULT build path. `AnalysisContextCollection` over the host
  roots (`lib` + local package libs). When `.dart_tool/package_config.json` is
  absent, `build` auto-falls-back to the syntax path with a one-line note;
  `--syntax` forces syntax; explicit `--resolved` with no package_config
  refuses (exit 66) with the `pub get` instruction.
- Implementation: `engine.buildResolved` (async) shares the whole post-parse
  tail (`_emitBuild`) with the sync syntax `build`; they differ only in how the
  file list is produced. `_extractUnit` extracts from either a syntax or a
  resolved `CompilationUnit`. Measured on codegraph's own repo: resolved and
  syntax builds produce a BYTE-IDENTICAL `code_graph.json` (extraction
  equivalence holds - Stage 2 is where element identity makes them diverge),
  26/26 files resolved, 0 fallbacks.
- Per file: `session.getResolvedUnit`; on `!is ResolvedUnitResult` OR any
  thrown resolution error, fall back to `parseString` for THAT file and count
  it. One stderr summary line: "N of M files fell back to syntax (resolution
  failed)" - same shape as the existing parse-error note.
- Per-file syntactic-error parity: resolved files flag the parse-error note
  only on `DiagnosticType.SYNTACTIC_ERROR`, NOT semantic errors - a file with
  an unresolved external `package:` import (absent dep) has a complete AST and
  must not be called "incomplete". (Learned in test: the fixture names go_router
  with no dep, which flagged 46 false "parse errors" until filtered.)
- Test: `test/resolved_build_test.dart` - `buildResolved` on the fixture (with a
  synthetic `.dart_tool/package_config.json` from `writeFixturePackageConfig`)
  produces a graph BYTE-IDENTICAL to the syntax build, and resolves files
  (146/146 on the fixture; external symbols left unresolved, self/SDK resolved).
- Refusal gate: no `.dart_tool` -> exit 66 with instruction, verified via CLI.
- REAL-APP VALIDATION on krdpass-app (1546 files, 243 providers, Flutter +
  Riverpod + go_router): resolved build 33s / 2.3GB RSS vs syntax 2.5s / 230MB -
  matches the viability prediction (under a minute; multiple GB is the daemon's
  constraint, not the one-shot build's). Resolved is deterministic across runs.
- BUG found + fixed via krdpass (resolved was LESS complete than syntax - a
  doctrine violation): GoRoute declarations were extracted only from
  `visitMethodInvocation`, which relied on the syntax parser MISPARSING
  `GoRoute(...)` as a method call. Under resolution it correctly parses as an
  InstanceCreationExpression, so the extractor never fired and resolved dropped
  every GoRoute-based `navigates-to` edge (39 on krdpass). Fixed by factoring
  `_recordGoRoute` and firing it from a new `visitInstanceCreationExpression`
  too; resolved now produces an IDENTICAL edge set to syntax on krdpass (0
  syntax-only, 0 resolved-only). Regression test: the fixture gains a resolvable
  fake `go_router` (in `.fixture_deps/`, outside scanned roots) so the
  byte-identical resolved-vs-syntax assertion now guards the constructor form;
  verified it fails when the new visitor is disabled.
  LESSON for Stage 2: other extractors that assume the syntax misparse
  (constructor-shaped calls seen as MethodInvocations) may need the same
  dual-form treatment. Audit the visitor for `looksLikeTypeName`/method-name
  constructor assumptions when element identity lands.

### Stage 2 - Element-checked extractors + the new model (the ceilings fall)
Each mechanism lands with a fixture case that syntax-only provably gets wrong;
keep the syntax path as the labeled fallback (tagged `heuristic`, vs `resolved`
for element-checked). This is where the wire-format redesign above lands
(`symbolId`, `confidence`).

CRITICAL fixture finding (2026-07-14): the current fixture uses `dynamic ref`
stand-ins and imports external packages (go_router) with no deps, so it CANNOT
exercise element-checked type detection - a `dynamic` receiver has no static
type to check. Stage 2's first task is therefore new, self-contained,
resolvable fixture material: a fake `Ref`/`WidgetRef` (declared in the fixture's
own package so it resolves against SDK+self with no external deps), a provider
value, and a WRAPPER object holding a Ref - the case syntax name-match misses.

- readers [DONE 2026-07-14]: `x.read(p)` is a reader edge when x's STATIC TYPE
  is (a subtype of) Ref/WidgetRef/ProviderContainer (`_isRefType`, checks the
  type + full supertype closure). Auto-adapts per unit: `node.realTarget?.
  staticType` is non-null on resolved units (type check fires), null on syntax
  units (name allow-list unchanged), so resolved is a strict superset - never
  fewer reader edges. Still gated on the arg resolving to a real provider, so no
  false edges. VALIDATED on krdpass: resolved catches 23 reader edges syntax
  misses (renamed params, `ProviderScope.containerOf(context).read(...)`,
  `ref.container.listen(...)`), loses 0; two spot-checked as true positives.
  Regression test: `test/resolved_build_test.dart` renamed-Ref case (syntax
  misses, resolved catches) via a resolvable fake `riverpod` Ref in the fixture.
- confidence column [DONE 2026-07-14]: `GraphEdge.confidence` =
  `resolved` | `heuristic` | `guessed` (NEVER-GUESS as data). Lean wire format:
  emitted only when not `heuristic` (absence = heuristic); `fromJson` defaults.
  Reader edges are tagged `resolved` when the receiver's static type is a Ref
  (threaded via `FileInfo.typedReaderKeys` = `<rel>|<provider>` set), else
  `heuristic`. Build prints an honesty metric: element-resolved reader fraction.
  VALIDATED on krdpass: syntax 0/1131 resolved; RESOLVED 1141/1154 (99%) - the
  13 heuristic remainders are receivers whose type didn't resolve, honestly
  flagged. Test asserts the caught renamed-Ref edge carries `confidence:
  resolved`. Other edge kinds (imports, subtype, nav) stay `heuristic`/`guessed`
  until their extractors go element-checked (the remaining ceilings).
- confidence SURFACED in queries [DONE 2026-07-14]: `readers`/`provider` text
  output marks heuristic reader edges `[unconfirmed]` + a legend, but ONLY in a
  resolved build (a syntax build is all-heuristic, so the marker would be noise
  - gated on the graph having any resolved edge). Closes the "confidence written
  but never read" gap. Validated on krdpass (marker on the `*Ref`-extension
  bare-call readers of connectivityProvider; absent in syntax). The actuator's
  "act only on resolved" gate is the rename refusal (3.1); the remaining
  consumer is a `uses`/`change` filter (deferred).
- callers/refs [DONE 2026-07-14, the flagship slice]: `callers|refs <Symbol>
  --resolved` resolves at query time and attributes each call/ref to its real
  target by `methodName.element`/`node.element` identity - `HomePage.build` vs
  `SettingsPage.build` instead of one lumped `build`. Opt-in (`--resolved`)
  because it does whole-context resolution (~30-40s on a large app; the "once in
  a while" refactor-safety case). Refuses without a package_config. VALIDATED on
  krdpass: syntax `callers build` = "16 sites, 984 declarations, matched by name
  across all" (useless for a refactor); resolved = 16 sites each attributed to
  its exact target. Text output lists a `targets:` summary + per-site `→ target`;
  `--json` carries `targets` + per-hit `target`. Test (codegraph_test.dart):
  two same-named `foo` methods on unrelated classes attribute to `A.foo`/`B.foo`.
  This is the actuator MVP: element-precise blast radius for safe refactoring.
  Callchain not yet converted (uses the same name-match path).
- INHERITANCE SAFETY [DONE 2026-07-14, actuator's read-only brain]:
  `callers/refs --resolved` also reports, per target, the supertype methods it
  overrides with their declaring library - the "what breaks through inheritance"
  half of the actuator, WITHOUT the risk of emitting edits. VALIDATED on
  krdpass: `callers build --resolved` flags `AutomaticKeepAliveClientMixin.build
  overrides State.build [package:flutter/...]` and `...Notifier.build overrides
  AsyncNotifier.build [package:riverpod/...]` (external = unsafe to touch) vs
  `UserCachedResourceNotifier.build overrides BaseCachedResourceNotifier.build
  [package:krdpass/...]` (project = rename together). An agent now knows, before
  touching a method, whether a signature change is safe and what else must
  change. Test: Derived.compute overrides Base.compute is reported. `--json`
  carries an `overrides` map. NEXT (deferred, high-stakes): edit EMISSION (rename
  → the patch), gated on this analysis refusing external-base overrides.
- impls/subtype [DONE 2026-07-14, confidence pass]: implements/extends edges are
  tagged `resolved` when the supertype NamedType carries a resolved element
  (`FileInfo.elementResolvedSupers`), `heuristic` under syntax. VALIDATED on
  krdpass: syntax 0/1708, RESOLVED 1708/1708 (100%) - every type-hierarchy edge
  element-confirmed, so `impls`/`change` can be fully trusted. Test asserts
  in-package supertype edges are resolved under resolution, 0 under syntax.
  STILL TODO (deeper impls win): use the element supertype CLOSURE to catch
  mixin applications / typedef aliases / inherited interfaces the text misses,
  and to disambiguate same-name supertypes at the edge TARGET (not just tag
  confidence). That is a follow-up; this pass did the confidence tagging.
- NOTE: Stage 2 broke the Stage 1 byte-identical invariant on purpose (resolved
  now adds confidence tags). The resolved-vs-syntax test was rewritten to assert
  equality MODULO confidence - same nodes/edges, only trust tags differ.
- nav resolution: route expressions resolve through constants/getters via
  element values; the three bespoke substitution mechanisms collapse.
- Acceptance: each flipped ceiling has a fixture that fails on the syntax arm
  and passes on the resolved arm, both in the same test. usefulness `--check`
  floors updated only for the deliberate truth changes, stated in the commit.

### Stage 3 - Warm daemon (latency + incrementality)
- `codegraph serve` holds the AnalysisContextCollection + graph resident;
  file-watch triggers incremental re-resolution (the analyzer owns
  invalidation). Query verbs become a thin unix-socket client with AUTOMATIC
  fallback to the one-shot path when no daemon runs. NOT MCP - no resident tool
  schema; agents shell out to the same CLI.
- MEASURE memory on the reference host before making the daemon a default
  (spike floor is ~800MB; a 1500-file host will be multiple GB). Record it here.
- Buys per-query latency (~220ms -> ~10ms) and the incrementality the one-shot
  design cannot retrofit.
- Acceptance: daemon and one-shot produce byte-identical graphs; killing the
  daemon mid-query degrades to one-shot with no wrong answer.

### Stage 4 - Benchmark verdict (does resolved actually win)
- Host suite + agent A/B compare resolved vs syntax arms. Resolved becomes the
  DEFAULT only if it wins on correctness without losing the zero-setup story
  (fallback keeps working with no `.dart_tool`).
- This stage is also where the north-star metric (BRD section 3) gets wired:
  the A/B is elevated from exploratory to a real gate on the task classes P0
  is meant to unlock.

## Risks (carried from viability + 2.1)

- analyzer version coupling to the host SDK constraints - test against the
  reference host's Flutter pin first; syntax fallback is the safety net.
- daemon memory (Stage 3) - measured floor ~800MB, multiple GB at host scale;
  gates the daemon default, not the build.
- determinism - resolved extraction must stay byte-deterministic (sorted
  element iteration) for check().

## Gates (every stage)
`dart analyze --fatal-infos && dart format --set-exit-if-changed . &&
dart test && dart test`, plus usefulness `--check`, perf compare, and the
reference host suite before release. Byte-stability on the reference host where
the stage touches the writer.
