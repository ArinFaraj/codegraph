# codegraph 0.8.0 — "Leverage" (design + implementation plan)

Theme: compound the shipped primitives into the three highest-value composites
for an AI-daily Flutter team: run only the tests a change can affect, score a
change's risk in one line, and make the Riverpod graph self-diagnosing. Plus
the one missing lookup verb (`tests <thing>`) and the pre-edit pack.

Depends on 0.6.0 (registry-resolved testRefs — MANDATORY for test selection)
and 0.7.0 (lint engine — risk score consumes it; Riverpod rules land on it).
Standing doctrine: plans/ROADMAP.md. One doctrine inversion applies here and
is the soul of this release: for anything that can SKIP tests or downplay
risk, uncertainty must FAIL OPEN (run more, score higher).

## Stage 1 — reverse test index + `tests <thing>` verb

The building block. Today testRefs is a count; selection needs the mapping.

- Do NOT store per-test mappings in the graph (bloat; the graph is
  lib-shaped). Instead: a verb-time scan — `scanTestIndex()` in a new
  lib/src/test_index.dart parses the test roots (same walk/closure code as
  `_scanTestRefs`; FACTOR the shared core out of engine.dart rather than
  duplicating it) and returns testFile -> Set<libFile referenced> (imports +
  export closure + closure-gated provider tokens mapped to declaring files).
  ~0.5s on the reference host — acceptable per-invocation cost, keeps the
  artifact clean.
- NEW verb `codegraph tests <file|provider>`: test files whose reference set
  contains the target (file arg via the shared resolveFileArg; provider arg
  maps to its declaring file(s) + token credits). Sorted, budgeted, --json.
  This answers the daily question the graph still can't: "which tests cover
  this?" — and gives `brief` its missing line (below).
- brief file cards append one line: `tests (N): <first 3> …` from the same
  index when N > 0, `tests: none` otherwise (an agent editing a file should
  see its coverage without asking).
- Tests: fixture coverage both directions (file arg, provider arg, none).

## Stage 2 — `affected-tests` (CI test selection)

- `codegraph affected-tests [--base <ref>] [--exec]`:
  1. changed set = diff machinery (same base resolution, same untracked
     handling);
  2. affected lib closure = changed ∪ transitive reverse-dependents
     (impact's dependentsOf iterated to fixpoint — unbounded depth; tests
     care about any path);
  3. affected tests = test files whose reference set (Stage 1 index)
     intersects the closure, PLUS test files that themselves changed.
- FAIL-OPEN GATES (each one prints its reason and outputs ALL test files):
  git missing/no base; graph missing or `format` mismatch; pubspec.yaml or
  any file under test roots matching `all_on` globs changed (default:
  `test/test_helpers/**`, `**/all_tests.dart`, `dart_test.yaml`,
  `pubspec.*`); zero-affected result when the changed set is non-empty AND
  contains lib files (an empty intersection with real lib changes is more
  likely an index blind spot than a truly isolated change — print all,
  state why; config `trust_empty: true` in codegraph.json's
  `affected_tests` section opts into trusting it later).
- Output: sorted test file list; `--exec` prints a ready
  `flutter test <files>` line (or `dart test` when no flutter dep —
  detectable from pubspec); `--json`.
- CI: init template gains a commented-out selective-test step (documented,
  off by default — turning CI test selection on is a host decision, not an
  init default).
- Tests: selection correctness on a fixture chain (change a leaf → its
  transitive dependents' tests selected, unrelated test not); every
  fail-open gate exercised; `--exec` shape.

## Stage 3 — `risk` (one-line change score) + diff integration

- `codegraph risk [--base <ref>]`: composite over the diff's data, three
  factors + lint, each reported with its evidence line:
  1. BLAST: size of the affected-lib closure (Stage 2's) — <10 LOW, <50
     MEDIUM, else HIGH;
  2. UNTESTED: any changed lib file (role view/controller/repository/
     provider) with zero Stage-1 test references → at least MEDIUM; more
     than 3 such files → HIGH;
  3. HOTSPOT: any changed file with in-degree ≥ 50 → at least MEDIUM; ≥ 150
     → HIGH;
  4. LINT: any NEW lint violation (vs baseline) → HIGH.
  Overall = max. Output: `risk: HIGH` + one line per contributing factor,
  nothing for quiet factors. Thresholds are constants documented in the
  README (fixed, boring, arguable — deliberately not configurable in v1;
  add config only when a real host needs it).
- diff card header gains the risk line; PR comment template unchanged (it
  already posts the diff card).
- Tests: each factor triggered in isolation on fixtures; max-combination.

## Stage 4 — Riverpod health as lint rules 5-6 + release

- Rule 5 `provider-cycle`: SCC (iterative Tarjan — recursion depth is a real
  risk on a 1.5k-node graph) over the provider-file dependency graph: edge
  A→B when a provider declared in file A has its declaring file
  watch/read/listen a provider declared in file B. HONEST LABELING: these
  are file-level cycles ("provider files X↔Y depend on each other"), not
  proven provider-level cycles — the output says so. Multi-provider files
  make this a candidate, which is exactly what the baseline ratchet is for.
- Rule 6 `god-provider`: reader count (watch+read+listen consumers) ≥
  config `god_provider_readers` (no default — rule off unless configured;
  the reference host would start at 100 and see translationsProvider at ~198, which is
  a TRUE positive worth having in the baseline).
- Both ride the 0.7.0 lint engine: same Violation shape, baseline, budget,
  CI behavior. No new machinery.
- Fresh-eyes review priorities: fail-open completeness in affected-tests
  (the ONLY feature in the roadmap that can eat a real signal — attack it
  hardest: renames, deleted tests, tests importing via package: self-name,
  integration_test-only coverage, the trust_empty path), risk-factor
  determinism, Tarjan on self-loops/isolated nodes, rule 5 false-positive
  labeling.
- Release 0.8.0 + reference-host rollout: enable rules 5-6 in codegraph.json with
  the baseline re-ratcheted; run affected-tests against a real recent branch
  and REPORT measured selection ratio (X of 328 test files) + wall-clock
  saved in the release notes; leave the CI selective step off pending a few
  weeks of `affected-tests` output being eyeballed in PR comments (add the
  list to the diff card comment as an info line instead — trust ramp).

## Explicitly out

- Auto-running the selected tests from the hook or any implicit path — test
  execution stays a human/CI decision; the tool only selects.
- Coverage-data integration (lcov) — different data source, different tool.
- Cross-package test selection for melos-style monorepos — revisit with the
  monorepo work if 0.6.0's self-location shows demand.
