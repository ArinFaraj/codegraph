# codegraph 0.7.0 — "Lint" (design + implementation plan)

Theme: the graph becomes PRESCRIPTIVE. Host repos (the host's AGENTS.md is the
motivating case) state architecture rules in prose that nothing enforces —
no cross-feature imports, layer ordering, banned provider kinds, provider
placement. The graph already contains every fact needed to check all of
them. `codegraph lint` turns the prose into a CI gate, with a baseline
ratchet so adoption on a repo with existing violations is a one-command
event, not a cleanup project.

This is the highest-leverage feature in the roadmap for the daily
"AI writes, humans review" workflow: it catches agent drift mechanically,
before review.

Ships after 0.6.0. NOTE: the pinned `line` key on `imports` edges was DEFERRED
out of 0.6.0 (capturing it means threading import-directive line numbers through
`FileInfo.internalImports`, which 0.6.0 had no other consumer for). It is now
this release's **Stage 1 first task**: add `line` to `imports` edges
(additive, `stats.format: 4`) before the lint rules that render `file:line`
violations. Standing doctrine: plans/ROADMAP.md.

## Design decisions (locked here)

- **Config = `codegraph.json` at the host package root.** stdlib jsonDecode,
  zero new deps (yaml is transitive-only; JSON is agent-friendly). Absent
  file = conservative defaults (below). Unknown keys → one warning line,
  ignored (forward compat).
- **A rule is a function, not a framework**: `List<Violation> Function(Graph,
  LintConfig)` plus an id string, held in a plain list. Violation =
  {rule, from (file id), to (file id or detail string), line?}. No plugin
  system, no rule registry indirection.
- **Baseline ratchet**: `codegraph lint --write-baseline` writes
  `docs/maps/lint-baseline.json` — the SORTED list of current violation
  identities. `lint` fails (exit 1) only on violations NOT in the baseline;
  fixed ones are reported as `stale baseline entries (N) — rerun
  --write-baseline to ratchet down`. Identity = `rule|from|to` (NO line
  numbers — moving code must not churn the baseline). Deterministic file, no
  timestamps; `build` never writes it, so the check() gate is unaffected
  (same reasoning as notes/, but this file IS generated — only ever by the
  explicit flag, which is the difference).
- **Never-guess applied to lint**: a rule fires only on facts the graph
  states (an import edge, a provider kind, a role). No heuristic "probably a
  violation" findings. Anything uncertain is simply not a violation.

## Rule set v1

1. `cross-feature-import` — config `features`: list of glob-ish prefixes
   (default `["lib/features/"]`; a prefix matches its immediate subdirs as
   feature units). Violation: an import edge between files under two
   DIFFERENT feature units of the same prefix. `allow`: list of
   `"from-unit -> to-unit"` exemption pairs.
2. `layer-order` — uses the existing role classifier. Config
   `layers.forbid`: list of `"roleA -> roleB"` strings; default:
   `repository -> view`, `repository -> widget`, `repository -> controller`,
   `state/model -> view`, `state/model -> widget`, `state/model ->
   controller`. Violation: an import edge whose src role → dst role is
   forbidden.
3. `banned-provider-kind` — config `banned_provider_kinds` (default `[]`;
   the reference host will set `["StateProvider", "StateNotifierProvider",
   "ChangeNotifierProvider"]`). Violation: a provider node of that kind.
   `to` = the kind, `line` = declaration line.
4. `provider-placement` — config `provider_homes`: list of roles a
   provider-declaring file must have (default: absent = rule off; the reference
   host
   sets `["provider", "controller", "application", "repository", "logic"]`).
   Violation: a declares edge whose src role isn't in the list.

Each rule's output section: header, sorted `from -> to (file:line)` lines,
capped by the shared budget. `--json` with the standard envelope. Exit codes:
0 clean-or-baselined, 1 new violations, 66 no graph.

## Stages

### Stage 1 — `imports.line` wire (deferred from 0.6.0) + config + violation model + rules 1-2

FIRST, land the deferred wire addition: `imports` edges gain an optional final
key `line` (int, 1-based line of the import/export/part directive; first
occurrence wins for duplicate imports of the same target). This means
`FileInfo.internalImports` (today a `List<String>` of target paths) must also
carry the directive line — thread it through `_parseFile`'s directive walk and
the `_writeGraph` emission. Bump `stats.format` to `4` (additive; `Graph.load`
already warns on skew). Byte-stability: the graph changes (import edges gain
`line`), so re-baseline the reference-host hash after this task; determinism +
round-trip tests must still pass. THEN:

lib/src/lint.dart: LintConfig.fromJson (with defaults), Violation, the rule
list with `cross-feature-import` and `layer-order`, text + JSON rendering,
budget discipline. bin dispatch + usage. Tests: fixture gains two feature
dirs with a crossing import + a repository→view import; config parsing
defaults; allow-pair exemption; determinism of output ordering.

### Stage 2 — rules 3-4 + baseline ratchet

Provider-kind and placement rules; `--write-baseline`; ratchet semantics
(new-only failures, stale-entry reporting); baseline file determinism test
(write twice → identical bytes); identity excludes line numbers (move a
violating import to a different line in the fixture → still baselined).

### Stage 3 — integration: CI, templates, host config

- init `--ci` workflow template gains a lint step after check:
  `dart pub global run codegraph:codegraph lint` (fails PR on new
  violations). init writes nothing else new — codegraph.json is authored by
  the host, but `init` prints a NOTE with a starter config when the file is
  absent and `lib/features/` exists.
- CLAUDE/skill/cursor `_commandBlock`: add
  `codegraph lint               # architecture rules — run before committing`.
- diff card: append a `lint:` line when new violations exist in the working
  tree (reuses the lint run; verb-only).
- doctor: checks codegraph.json parses when present.
- Tests: template greps; diff-card line appears in a violating fixture.

### Stage 4 — fresh-eyes review + release + reference-host adoption

Review brief priorities: false-positive hunt per rule (feature-unit
derivation on Windows-style separators? packages/ paths crossing the
features prefix? role misclassification making layer-order fire wrongly —
if role data is too coarse for a rule, the rule must narrow, not guess),
baseline identity collisions, determinism, budget/JSON caps. Then: version
0.7.0, CHANGELOG, release. reference-host rollout: author `codegraph.json` (the
AGENTS.md non-negotiables translated: features = lib/features/ + the legacy
roots as allow-listed exceptions; banned kinds per Standard 03),
`lint --write-baseline`, commit baseline, enable the CI step, add one
LIMITATIONS/notes line telling agents the rules are now mechanical. Report
the reference host's initial violation counts per rule in the release notes — that
number is the feature's pitch.

## Explicitly out (this release)

Riverpod health rules (cycles, god providers) — 0.8.0 adds them as rules 5-6
on this engine. Auto-fix (`lint --fix`) — never speculative; revisit only
with evidence. Custom user-defined rule expressions — the four built-ins
cover the stated standards; a rule DSL is framework-bait.
