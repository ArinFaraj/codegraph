# codegraph 0.8.0 — "Upgrade & Plan" (design + implementation plan)

**Re-scoped 2026-07-03** on user steer: the tool's primary job is to help an AI
UNDERSTAND, NAVIGATE, FIND, PLAN, and CREATE-NEW-FEATURES-FROM-EXISTING on a
production host. Lead with that + the missing upgrade story. The original 0.8.0
(affected-tests / risk / CI test-selection) was CI-leverage — deferred to
`0.9.0-leverage.md` unless a real need appears. Every feature here is
dogfooded on a reference host before it ships; anything that duplicates `brief` without
adding real planning value is cut.

Standing doctrine: plans/ROADMAP.md.

## The problem this release closes

1. **No upgrade path.** The graph auto-migrates (gitignored, rebuilt each
   session by the hook / in CI by `check`; format skew handled by
   `Graph.load`). But the SCAFFOLDING (`.claude/skills/code-map/SKILL.md`,
   CLAUDE.md block, hook, `.cursor/rules/codegraph.mdc`, CI workflow) is
   generated ONCE by `init`, which SKIPS existing files. New verbs (`lint`,
   `doctor`, `brief`…) never reach a host that ran `init` at an older version,
   and nothing tells the AI its skill/command-list is stale. Fix: version-stamp
   the scaffolding, add `codegraph upgrade`, and make `doctor`/`passport` nudge
   when the stamp is behind the binary.
2. **Planning is under-served.** `brief` shows what a feature IS; nothing helps
   an AI plan a NEW feature modeled on an existing one (the host project's whole
   `feature` skill is "copy `lib/features/sign_in_with_oauth/`"). Fix: a
   `blueprint` primitive that extracts an exemplar feature's STRUCTURE (files by
   role, provider-wiring topology, routes, external deps, tests) as a build
   plan — dogfooded against the real exemplar before shipping.

## Stage A — version-stamped scaffolding + `upgrade` + skew nudge

- **Stamp** every generated artifact with the writing binary's version:
  CLAUDE.md block marker becomes `<!-- codegraph:begin vX.Y.Z -->`; hook / skill
  / cursor rule / workflow each carry a `codegraph-scaffold: vX.Y.Z` comment
  line (in a spot that survives being read back — frontmatter comment for the
  skill, shell comment for the hook, YAML comment for the workflow). Thread
  `version` into `_commandBlock`/`_hook`/`_skill`/`_cursorRule`/`_workflow`.
- **`codegraph upgrade`** (new verb): re-generate the codegraph-OWNED files in
  place — hook, skill, cursor rule (only if already present), workflow (only if
  already present) — OVERWRITING them to the current version; and replace the
  CLAUDE.md block BETWEEN its markers (preserving all user content outside).
  Never touches host-owned files: `.claude/settings.json`, `docs/maps/
  LIMITATIONS.md`, `codegraph.json`, `docs/maps/notes/`. Prints what refreshed +
  a reminder to `codegraph build`. Idempotent: re-running when already current
  reports "already vX.Y.Z" and changes nothing (byte-identical).
- **`doctor` check**: read the scaffolding version (from the CLAUDE marker; fall
  back to the hook stamp) vs the binary version. Behind → fail/note with fix
  `codegraph upgrade`. Missing stamp (pre-0.8 install) → note "run codegraph
  upgrade to version-stamp".
- **`passport` nudge**: when scaffolding < binary, prepend one line
  `codegraph: skills are vX (binary vY) — run 'codegraph upgrade' to refresh`.
  This is the mechanism that TELLS THE AI at session start (the hook runs
  passport every session). Silent when current or when no scaffolding exists.
- Never-guess: if the stamp can't be found/parsed, treat as "unknown" and nudge
  to upgrade rather than guessing it's current.
- Tests: stamp present in every generated file; `upgrade` refreshes a stale
  skill/CLAUDE block while preserving surrounding CLAUDE.md content and NOT
  touching settings/LIMITATIONS/codegraph.json; idempotent re-run; doctor +
  passport skew detection both directions (stale → nudge, current → silent).
- **Reference-host dogfood**: upgrade the host scaffolding, diff it, confirm the
  skill/command-list now lists lint/doctor and user content is intact.

## Stage B — `blueprint <feature>` (planning primitive)

**Dogfood done (2026-07-03) — VERDICT: BUILD IT.** Planning a new feature from
`lib/features/sign_in_with_oauth/` with today's verbs: `brief <feature>` ranks
the CURRENT state by importance (in-degree, reader counts, the cross-area seam) —
excellent for understanding, but for CREATING an analogous feature it is missing
exactly three graph-derivable things: (1) LAYERED BUILD-ORDER (the exemplar is
`domain → data → application → presentation → routing`, but brief sorts by
in-degree, not layer); (2) the WIRING TOPOLOGY abstracted ("controller watches
repository-provider + auth-service; repository-provider reads dioFactory +
cookieJar" — the reusable pattern, not a ranking); (3) NAMING CONVENTIONS as a
template (`X_repository.dart` + `X_repository_provider.dart`, `X_controller.dart`,
`X_failure.dart`, `X_routes.dart`). None duplicate brief — blueprint is additive.

`codegraph blueprint <feature-dir>` outputs a build-order plan derived from the
exemplar —
- files grouped by ROLE in dependency order (repository → provider/controller →
  page → widget → routing → test), each with its symbols;
- the provider-wiring TOPOLOGY (which roles declare providers, what they
  watch/read, which are exported vs internal);
- routes the feature registers (from navigates/route data);
- EXTERNAL providers/deps it consumes (the cross-area seam a new feature must
  wire to — reuses `_external`);
- its test files + what they cover (reuses testRefs).
Output = a checklist an AI follows to scaffold an analogous feature, plus the
naming conventions observed (`_page.dart`, `_controller.dart`, `_repository.dart`,
`xProvider`). `--json` for a `feature` skill to consume.
- Never-guess: report only structure the graph states; do NOT invent files the
  new feature "should" have beyond the exemplar's own roles.
- Cut criteria: if the dogfood shows `brief <feature>` + the existing `feature`
  skill already give an AI everything it needs, DON'T ship blueprint — instead
  improve `brief`'s feature card with the one or two missing facts. Usefulness
  over surface area.
- Tests + reference host: run `blueprint lib/features/sign_in_with_oauth` and confirm
  the output is a plan a fresh agent could actually build from.

## Stage C — review + release + reference-host rollout

Adversarial review (upgrade must never clobber user content; blueprint must not
guess). Fix round. Version 0.8.0, CHANGELOG. Reference host: run `upgrade` for real
(refresh the stale scaffolding), report. Update memory + LIMITATIONS.

## Explicitly deferred (to 0.9.0)

- `affected-tests` / `risk` / CI test-selection (CI-leverage, not the primary
  understand/plan job). `tests <thing>` (which tests cover X) may ride Stage B
  if the dogfood shows planning needs it.
- Riverpod cycle / god-provider lint rules (health, mild value) — add on the
  0.7.0 lint engine when there's a real ask.
