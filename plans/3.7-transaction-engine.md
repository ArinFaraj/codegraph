# 3.7 - the transactional change engine ("the tree never ends red")

Status: planned 2026-07-21. The next major arc after the campaign-v2 verdict.

## Why this is the groundbreaking bet

Every measurement this project has produced points at one agent failure mode:
agents leave working trees broken. Campaign v2 proved the counter-thesis for
renames - the treatment arm hit 100% safety because rename is TRANSACTIONAL
(prevalidate, stage, apply, rollback) and REFUSES what it cannot prove. The
bet: generalize that guarantee from renames to ANY change.

    codegraph tx --patch <file|-> [--cmd '<shell command>']
      1. snapshot   - record tree state (git stash-like, tx-owned)
      2. apply      - the patch/command runs
      3. analyze    - dart analyze (scoped to the change's closure)
      4. test       - affected-tests plan, executed
      5. green      -> keep; emit an evidence card (what ran, why, results)
         red/throw  -> ROLLBACK to the snapshot; emit what broke and where

The agent contract becomes: propose any change; the tree ends green or
untouched, with evidence either way. That converts codegraph from "safe
renames" into a safety substrate for ALL agent edits - including future
signature changes, conformance fixes, and LLM-proposed refactors.

## Why it is buildable now (composition, not invention)

- Atomic staged apply + rollback + retained-recovery reporting: shipped (3.2).
- affected-tests with fail-open expansion and an executable mutation oracle:
  shipped (3.3).
- Cancellation that cannot interrupt the critical section: shipped (3.4 line).
- The agent-impact harness to MEASURE it: shipped, with a proven mining loop.

## Honesty constraints (non-negotiable, from the standing doctrine)

1. "Green" claims exactly what was proven: which analyze scope, which suites,
   why those (the affected-tests evidence card), and whether fail-open
   expanded. If the full suite did not run, the card says so.
2. Rollback is transaction-like, never claimed atomic across files (the 3.2
   contract); an interrupted install still reports recovery paths.
3. `tx` never invents fixes. Red means rollback + report, not retry-mutate.
4. Timeouts/cancellation observe the critical-section rules (exit 130 only at
   safe points; never a half-applied tree without a recovery report).

## Stages

1. **Snapshot/restore core.** Tx-owned snapshot (copy-on-write of tracked +
   untracked-but-not-ignored files, or git worktree stash when available);
   restore proven byte-exact under injected mid-apply failures. No reliance on
   the host's git state being clean.
2. **Pipeline.** patch/cmd -> analyze -> affected-tests -> keep/rollback, with
   the evidence card (text + `--json`). Wire cancellation checkpoints between
   phases; the restore path is the critical section.
3. **Harness measurement.** New benchmark tasks where baseline agents
   historically end red (multi-file edits with a hidden breaking consumer);
   measure tree-ends-red rate with vs without `tx`. Pre-registered gate:
   treatment tree-ends-red must be 0; adoption is part of what is measured
   (the scaffolding teaches: route risky multi-file edits through tx).
4. **Scaffolding + docs.** The CLAUDE block gains rule (3): "for multi-file or
   uncertain edits, run them inside `codegraph tx`". LIMITATIONS seeded with
   what tx does not prove (runtime behavior, unselected suites when targeted).

## Out of scope (this arc)

- Auto-fix/retry loops on red (a later, separate bet with its own gates).
- Runtime verification (`codegraph observe` - the VM-service truth layer is
  its own plan).
- Signature-change generation (still gated on production-scale value evidence;
  tx makes it SAFER to attempt when unlocked, not sooner).

## Exit criteria

`tx` applies real multi-file patches on the reference host with byte-exact
rollback under fault injection; the evidence card is honest under fail-open;
the harness shows tree-ends-red = 0 in the treatment arm on the new task
class; scaffolding teaches it and the guard test pins it.
