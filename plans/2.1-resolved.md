# codegraph 2.1 - "Resolved" (element identity, warm daemon, doctrine amendment)

The 2026-07-11 audit's central verdict: every ceiling the project documents
about itself (nav resolution stuck at 39/99 after three mechanisms, wrapper
and renamed-parameter ref reads invisible, callers/callchain name-matched
across unrelated same-named declarations) traces to ONE decision - syntax-only
parsing - and that decision is the only rejection in the changelog with no
reopening condition. 2.1 reopens it deliberately.

## Doctrine amendment (replaces ROADMAP standing doctrine item 1)

OLD: "Syntax-only parsing; no pub get in the host. Ever."
NEW: "Syntax-only is the DEFAULT and the zero-setup fallback; resolved
analysis is an opt-in layer that requires the host's own `pub get` to have
succeeded (a constraint the host's CI already accepts for its tests). A
resolved answer must never be less honest than a syntax answer: where
resolution fails for a file, that file falls back to syntax extraction and
the output says so."

## Why element identity fixes four ceilings at once

The extractor already uses package:analyzer ASTs. A resolved unit is the SAME
AST with staticType/staticElement populated - the visitors stay, the checks
get stronger:
- readers: `x.read(p)` is a reader edge when x's static type is
  Ref/WidgetRef/ProviderContainer - kills the receiver allow-list, catches
  wrapper objects, renamed parameters, and rebound tear-offs by construction.
- callers/refs/callchain: match methodName.staticElement identity against the
  declaration's element - same-named methods on unrelated classes stop
  polluting counts; the multi-declaration note becomes exact partitioning.
- nav resolution: route expressions resolve through constants/getters via
  element values instead of three bespoke substitution mechanisms.
- impls: mixin applications, typedef aliases, and inherited interfaces come
  from the element model's supertype closure, not text.

## Stages

1. **Resolution harness**: `AnalysisContextCollection` over the host root
   (requires .dart_tool/package_config.json). `build --resolved` flag; per
   file, `session.getResolvedUnit`; on any resolution failure, fall back to
   parseString for that file and count it (stderr summary, same shape as the
   parse-error note). Fixture tests need a synthetic package_config.json in
   the temp dir - write it in writeCodegraphFixture (paths are known).
   Measure cold/warm build time on the reference host and record it in this
   plan before proceeding (expect minutes cold, tens of seconds warm; that is
   what makes stage 3 necessary).
2. **Element-checked extractors**: readers receiver-type check, callers
   element identity (keep name-match as the syntax fallback), impls via
   element supertypes. Each mechanism lands with a fixture case that
   syntax-only provably gets wrong (the wrapper-ref xfail in the host suite
   flips to XPASS here - promote it in the same commit).
3. **Warm daemon**: `codegraph serve` holds the AnalysisContextCollection +
   graph resident; file-watch triggers incremental re-resolution (the
   analyzer does the invalidation); query verbs become a thin client (unix
   socket) with automatic fallback to the current one-shot path when no
   daemon is running. NOT MCP - no resident tool schema, agents still shell
   out to the same CLI; this spends process memory to buy per-query latency
   (~220ms -> ~10ms) and unlocks incrementality the one-shot design cannot
   retrofit.
4. **Benchmark verdict**: the host suite + A/B harness compare resolved vs
   syntax arms; resolved becomes the default only if it wins on correctness
   without losing the zero-setup story (fallback keeps working with no
   .dart_tool).

## Risks to respect
- analyzer version coupling to the host's SDK constraints (the collection
  resolves against the host's package config; test against the reference
  host's Flutter pin first).
- Memory: resolved element models for 1.5k files will be hundreds of MB in
  the daemon; measure before defaulting.
- Determinism: resolved extraction must stay byte-deterministic for the
  check() gate (element iteration order sorted, as the syntax path already
  learned the hard way).
