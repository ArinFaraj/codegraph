---


Implements the actuator's write half (BRD pillar P2). Builds directly on 3.0's
element-precise callers + inheritance-safety analysis: `codegraph rename
<Symbol> <newName>` emits the edit set to rename every element-resolved
reference, gated so it refuses anything it cannot do completely and safely.

This is the advisor->actuator turn made real: not "here are the call sites" but
"here is the exact patch, and I refuse if it would break the build."

## Safety doctrine (NEVER-GUESS, sharpened for a mutation)

A wrong rename breaks the build - far worse than a missing one. So the actuator
REFUSES unless the rename is provably complete and safe. Every gate below is a
refusal with a clear reason, never a best-effort guess:

1. UNIQUE TARGET. Resolve `<Symbol>` (bare name or `Class.method`) to exactly
   one declaration element. Multiple unrelated declarations of the name ->
   refuse, list them, ask the user to qualify (`Class.method`).
2. NO OVERRIDE UP. If the target overrides a supertype method (3.0's
   `_overridesOf`): external base (dart:/package:flutter/...) -> refuse
   ("cannot rename framework API"); project base -> refuse for the MVP
   ("overrides a project method; whole-hierarchy rename is a later slice").
3. NO OVERRIDE DOWN. If a project subtype overrides the target (graph subtype
   edges whose child class declares the same name) -> refuse (same reason).
4. COMPLETE RESOLUTION. Every file with a textual occurrence of the name must
   resolve. Any resolution fallback -> refuse (cannot guarantee we saw every
   reference). Requires the host's package_config; refuse without it.
5. NAME SANITY. `<newName>` is a valid Dart identifier and does not already
   exist as a conflicting member in the same scope (best-effort: refuse on an
   obvious collision).

What passes all gates: unambiguous standalone methods, top-level functions, and
private members - the set where element-resolved references ARE the complete set.

## Behaviour

- Default: emit a unified diff (declaration site + every element-resolved
  reference's name token rewritten to `<newName>`). Print the count and the
  target's identity. Do NOT touch the working tree.
- `--apply`: write the edits to disk (reversible via git). Explicit only - the
  BRD non-goal forbids silent application.
- `--json`: structured edits `[{file, line, offset, length, old, new}]` + the
  refusal reason when refused.
- Slow (query-time whole-context resolution, tens of seconds) - the "once in a
  while" refactor case, same as `callers --resolved`.

## Mechanism

Reuse 3.0's resolved harness. Resolve all graph files once. Find the target
declaration element(s) by visiting declarations named `<method-part>` and taking
`declaredFragment.element`; run the gates. Then collect every reference whose
resolved element IS that target element (identity within the one
AnalysisContextCollection is canonical), recording each name token's
`offset`/`length`. Apply the rewrites per file (descending offset so earlier
edits don't shift later ones).

## Acceptance

- Fixture: rename a private/standalone method -> diff rewrites decl + all call
  sites; `--apply` makes the file compile with the new name; a same-named method
  on an unrelated class is UNTOUCHED (element identity, not name).
- Refusal tests: overrides-framework -> refuse; ambiguous name -> refuse+list;
  no package_config -> refuse.
- krdpass dry-run: pick a safe private helper, confirm the diff is complete and
  correct; confirm `rename build newBuild` REFUSES (framework override).

## Gates
dart analyze --fatal-infos, dart format, dart test, plus the krdpass dry-run.

## Out of scope (later)
- Whole-hierarchy rename (base + all overrides together).
- Signature changes (arg add/remove/reorder) - a distinct, harder actuator.
- Cross-package public-API renames where consumers are out of the graph's scope.
